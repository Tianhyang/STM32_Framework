# STM32F103RCT6 Project Log

## Project Goal

Build a production-oriented STM32 project from scratch for `STM32F103RCT6` with:

- Clear separation between platform layer and application layer
- Future FreeRTOS integration
- Room for C++ driver encapsulation
- A learning path that can still start from simple LED bring-up

## Step 1: Create Project Skeleton

Purpose:

- Define a clear layered directory structure before writing code
- Reserve locations for board support, chip support, RTOS, drivers, and application code
- Make later chip migration easier by avoiding app-layer dependence on chip details

What was done:

- Created the main project folders under `app`, `platform`, `drivers`, `rtos`, `toolchain`, `docs`, `third_party`, `build`, and `output`
- Reserved board-specific, chip-specific, CMSIS, HAL, startup, and future C++ wrapper directories

## Step 2: Prepare Vendor Materials

What was done:

## Step 9: Add USART1 Serial Bring-Up

Purpose:

- Establish a simple serial debug path through the onboard CH340
- Make it easier to observe task startup and runtime state
- Keep the serial path layered so it can evolve into a richer logger later

What was done:

- Added board mapping for `USART1`, `PA9` TX, `PA10` RX, `115200` baud
- Added UART HAL support to the project
- Added a generic UART driver interface
- Added a STM32 BSP UART implementation using blocking transmit
- Added boot logs and LED task heartbeat logs over `USART1`

Design note:

- This is intentionally the simplest reliable serial path
- It uses polling transmit first so bring-up is easy to validate before moving to DMA, queues, or a dedicated logger task

Clock note:

- The imported `system_stm32f1xx.c` had `SystemCoreClock` initialized to `16000000`
- Before explicit board clock setup, reset startup can leave the MCU running from the default internal clock path
- That mismatch caused UART baud calculations to be wrong during earlier bring-up
- The project now uses an explicit board clock configuration path instead of relying on the startup default

## Step 10: Add Lightweight Logging and Board Clock Setup

Purpose:

- Replace scattered raw UART prints with a cleaner application logging entry
- Move the board from implicit default clock startup to an explicit clock policy
- Keep bring-up robust by falling back if external clock startup fails

What was done:

- Added a simple `log_info()` interface backed by `USART1`
- Switched application boot and LED heartbeat output to use the log service
- Added board-level clock initialization
- The board now first attempts `HSE 8MHz -> PLL x9 -> 72MHz`
- If HSE startup or PLL configuration fails, it falls back to the default HSI clock path

Design note:

- The logging layer is intentionally small and synchronous for now
- The board clock policy is explicit, but still conservative because it includes a fallback path

## Step 16: Split Application Init into Services and Tasks

Purpose:

- Make the startup path easier to understand and extend

## Step 17: Introduce a Dedicated IRQ Routing Layer

Purpose:

- Keep vector entry functions thin and predictable
- Prevent `stm32f1xx_it.c` from turning into a mixed dump of IRQ logic
- Create a stable place for future peripheral interrupt expansion

What was done:

- Added `board_irq.c/.h` as the board-level IRQ routing module
- Moved actual IRQ handling logic out of `stm32f1xx_it.c`
- Kept `stm32f1xx_it.c` as a thin vector wrapper layer only
- Added routing for:
  - core faults
  - `SysTick`
  - `USART1`
  - `EXTI0`
  - `EXTI15_10`

Design note:

- Keys are still polled in the current framework
- EXTI handlers are added now as routing scaffolding so switching keys or GPIO inputs to interrupt mode later will not require restructuring the vector layer

## Step 18: Prepare an EXTI-Based Key Wakeup Path

Purpose:

- Keep the key framework responsive without moving debounce and event publishing into interrupt context
- Reuse the new IRQ routing layer for a real peripheral input path
- Build a safer bridge between hardware interrupts and application tasks

What was done:

- Added a key driver mode concept for polling or interrupt-based GPIO configuration
- Added a key IRQ callback registration path in the BSP key driver
- Configured `K1` and `K2` to use GPIO EXTI mode with rising/falling edge interrupts
- Connected EXTI callbacks to the key task through FreeRTOS task notifications
- Kept debounce, stable-state tracking, and event publishing inside the key task

Design note:

- This is intentionally not a "do everything in ISR" design
- EXTI is only used to wake the task early
- The actual key state filtering still happens in task context, which is a better fit for maintainability and future feature growth

## Step 19: Separate Key Wakeup Strategy from Key Event Logic

Purpose:

- Keep the key demo task focused on debounce and event generation only
- Avoid hard-coding EXTI notification details directly into application logic
- Prepare for future switching between polling, EXTI wakeup, or other scan strategies without rewriting the task

What was done:

- Added `app_key_input.c/.h` as a dedicated key input wakeup module
- Moved task notification and IRQ callback binding out of `app_key_demo.c`
- Centralized key mode selection and per-key initialization in the new module
- Kept the key task responsible only for state sampling, debounce, and event publication

Design note:

- This is a small but important framework cleanup step
- The application task now depends on a stable "input wait" interface instead of knowing whether wakeup comes from polling or interrupts
